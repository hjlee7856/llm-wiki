# LLM-Wiki 사용법

이 볼트는 “LLM을 이용해 지식을 축적하고 관리하는 개인 지식베이스”입니다.

핵심 아이디어는 단순합니다. 원본 자료를 질문할 때마다 다시 찾고 요약하는 대신, LLM이 원본을 읽고 지속적으로 갱신되는 마크다운 위키를 유지합니다. 사용자는 자료를 고르고 질문을 던지며 방향을 잡고, LLM은 요약, 연결, 색인, 충돌 표시, 기록 관리를 맡습니다.

이 위키는 검색을 대체하는 시스템이라기보다, 원본과 사용자 사이에 두는 `컴파일된 지식층`에 가깝습니다. 질문 시점마다 다시 조립하는 대신, 자주 쓰는 해석과 비교 결과를 파일로 남기고 계속 갱신하는 방식입니다.

참고한 패턴: [karpathy/llm-wiki.md](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)

## 세 계층

### 1. `raw`

원본 자료를 보관하는 곳입니다.

- 논문, 기사, PDF에서 뽑은 마크다운, 회의록, 메모, 이미지, 데이터 파일을 둡니다.
- 원본은 가능한 한 수정하지 않습니다.
- LLM이 읽을 수는 있지만, 원본 자체를 마음대로 고치지 않게 합니다.
- 원본은 “진실의 기준”입니다.

### 2. `wiki`

LLM이 유지하는 지식 위키입니다.

- 원본에서 뽑은 요약, 주제 페이지, 인물/조직/개념 페이지, 비교표, 종합 분석을 둡니다.
- 새 자료가 들어오면 기존 페이지를 갱신하고, 관련 페이지를 서로 링크합니다.
- 오래된 주장과 새 자료가 충돌하면 덮어쓰지 말고 충돌을 표시합니다.
- 사용자는 읽고 검토하며, LLM은 구조화와 유지보수를 담당합니다.
- 좋은 질문 결과와 분석도 `wiki/queries`에 다시 편입될 수 있습니다.

### 3. `AGENTS.md`

LLM에게 이 위키를 어떻게 다뤄야 하는지 알려주는 운영 규칙입니다.

- 자료를 ingest할 때 무엇을 갱신할지
- 질문에 답할 때 어떤 순서로 읽을지
- 색인과 로그를 어떻게 유지할지
- 원본과 파생 지식을 어떻게 구분할지

## 핵심 파일

### `wiki/index.md`

내용 중심 색인입니다.

- 위키에 있는 주요 페이지를 한 줄 요약과 함께 나열합니다.
- LLM은 질문에 답하기 전에 이 파일을 먼저 읽고 관련 페이지를 찾습니다.
- 새 페이지를 만들거나 큰 내용을 갱신하면 이 파일도 갱신합니다.

### `wiki/log.md`

시간순 작업 로그입니다.

- ingest, query, lint 작업을 날짜별로 기록합니다.
- 어떤 원본을 처리했는지, 어떤 위키 페이지가 바뀌었는지 남깁니다.
- 나중에 “최근에 무엇을 했는지”를 LLM과 사용자가 함께 파악하는 데 씁니다.

### `wiki/overview.md`

현재 위키가 다루는 지식의 큰 그림입니다.

- 주제 전체의 현재 종합 관점을 적습니다.
- 세부 페이지가 늘어나도 이 파일에서 큰 흐름을 확인할 수 있게 합니다.

### `wiki/ranking.md`

위키가 커졌을 때 어떤 페이지를 계속 활성 상태로 둘지 판단하는 기준입니다.

- 중요도, 신뢰도, 최신성을 기준으로 간단히 점수를 매깁니다.
- 원본 자료인 `raw`는 절대 가지치기하지 않습니다.
- 낮은 우선순위의 파생 페이지는 삭제보다 `wiki/archive` 이동을 우선합니다.
- archive 처리도 `wiki/log.md`에 기록합니다.

## 기본 작업 흐름

이 위키의 기본 흐름은 `raw -> wiki -> query -> lint`입니다.
즉 원본을 읽어 위키로 컴파일하고, 질문 결과 중 재사용 가치가 있는 것은 다시 위키 자산으로 남기며, 주기적으로 건강 상태를 점검합니다.

### 1. Ingest

새 자료를 `raw`에 넣고 LLM에게 처리하라고 요청합니다.

LLM은 다음 순서로 작업합니다.

1. 원본을 읽습니다.
2. 핵심 주장, 사실, 날짜, 이름, 숫자, 근거를 추출합니다.
3. `wiki/sources`에 원본 요약 페이지를 만듭니다.
4. 관련 주제나 엔티티 페이지를 새로 만들거나 갱신합니다.
5. 기존 내용과 충돌하는 부분이 있으면 표시합니다.
6. `wiki/index.md`를 갱신합니다.
7. `wiki/log.md`에 ingest 기록을 추가합니다.

### 2. Query

위키를 대상으로 질문합니다.

LLM은 다음 순서로 답합니다.

1. `wiki/index.md`를 먼저 읽습니다.
2. 관련 페이지를 찾아 읽습니다.
3. 필요하면 원본 자료도 확인합니다.
4. 답변에는 어떤 위키 페이지나 원본을 근거로 삼았는지 표시합니다.
5. 가치 있는 답변이면 `wiki/queries`에 새 분석 페이지로 저장합니다.
6. 저장했다면 `wiki/index.md`와 `wiki/log.md`를 갱신합니다.

질문 결과를 파일로 남기는 이유는, 좋은 해석이 대화창에서 사라지지 않게 하기 위해서입니다.

### 3. Lint

주기적으로 위키 상태를 점검합니다.

확인할 것:

- 링크되지 않은 고아 페이지
- 색인에 빠진 페이지
- 같은 내용을 다르게 말하는 중복 페이지
- 새 자료와 충돌하는 오래된 주장
- 출처가 없는 강한 주장
- 자주 등장하지만 아직 독립 페이지가 없는 개념
- 보강할 만한 질문이나 추가 자료 후보

### 4. Prune

위키가 너무 커지면 낮은 우선순위 페이지부터 활성 색인에서 제외합니다.

LLM은 다음 순서로 작업합니다.

1. `wiki/ranking.md`의 기준을 읽습니다.
2. `wiki/index.md`와 각 페이지의 frontmatter를 확인합니다.
3. `importance + confidence + freshness`가 낮은 페이지를 후보로 표시합니다.
4. 원본 `raw`는 절대 삭제하거나 이동하지 않습니다.
5. 삭제 대신 `status: archived`로 바꾸고 `wiki/archive`로 이동합니다.
6. 관련 페이지와 `wiki/index.md`에서 활성 링크를 정리합니다.
7. `wiki/log.md`에 archive 기록을 추가합니다.

## 사용 예시

### 새 글을 넣을 때

```text
raw/articles/example.md를 ingest해줘.
원본 요약을 만들고, 관련 topic/entity 페이지를 갱신하고,
index와 log도 업데이트해줘.
```

### 질문할 때

```text
이 위키 기준으로 X와 Y의 차이를 비교해줘.
먼저 index를 보고 관련 페이지를 찾은 뒤,
근거 링크와 함께 답해줘.
가치 있으면 wiki/queries에 저장해줘.
```

### 점검할 때

```text
wiki lint를 실행해줘.
고아 페이지, 충돌, 오래된 주장, 출처 없는 주장,
새로 만들면 좋을 topic/entity 후보를 찾아줘.
```

## 운영 원칙

- 원본과 LLM이 만든 요약을 섞지 않습니다.
- 확실하지 않은 내용은 확정문처럼 쓰지 않습니다.
- 새 자료가 기존 주장과 충돌하면 삭제보다 “충돌 기록”을 우선합니다.
- 위키 페이지는 짧고 연결 가능하게 유지합니다.
- 색인과 로그는 매 작업마다 갱신합니다.
- 자료가 많아지면 낮은 랭킹의 파생 페이지부터 archive 처리합니다.
- 사용자는 자료 선택, 검토, 질문, 방향 설정을 맡습니다.
- LLM은 요약, 연결, 갱신, 로그 작성, 위키 정리를 맡습니다.
- 작은 규모에서는 `wiki/index.md`와 구조화된 페이지 집합만으로도 많은 질문을 처리합니다.
- 규모가 커질 때만 별도 검색 계층이나 graph 계층을 추가로 검토합니다.

## 선택적 보조 검색 계층: MemPalace

이 저장소에서는 `MemPalace MCP`를 `작업 메모리 검색` 용도로만 붙입니다.

- 용도: 과거 작업 세션, 임시 결정 맥락, 아직 위키로 승격되지 않은 히스토리를 다시 찾기
- 비의도: `raw` 대체, `wiki` 대체, 최종 근거 저장소 역할
- 원칙: `MemPalace -> wiki/raw 검증 -> 필요한 내용만 wiki 반영`

프로젝트 설정 파일은 `mempalace.yaml`에 두고, 실제 로컬 palace 데이터는 `.mempalace/` 아래에 두되 버전 관리하지 않습니다.
직접 CLI로 확인할 때는 `mempalace --palace .mempalace/palace search "..."`처럼 palace 경로를 명시합니다.

## Next.js GUI

이 저장소에는 `codex cli`를 직접 백엔드로 호출하는 Next.js GUI가 저장소 루트의 `web/` 아래에 있고, 이 볼트는 `obsidian/` 아래에 있습니다.

- 목적: API 과금 대신 현재 로그인된 `codex cli` 세션을 통해 위키를 브라우저에서 탐색하고 질의
- 방식: Next.js Route Handler가 `codex exec`를 실행하고 마지막 응답만 받아서 UI에 표시
- 전제: `codex login`이 이미 끝나 있어야 함
- 구조: 저장소 루트에서 `web/`와 `obsidian/`를 분리하고, `web`은 `obsidian/`를 고정된 볼트 루트로 사용함

실행 예시:

```powershell
cd ../web
npm.cmd install
npm.cmd run dev:lan
```

브라우저에서 `http://localhost:3000` 또는 같은 네트워크의 `http://<your-ip>:3000`으로 접속합니다.

주의:

- 현재 스캐폴드는 인증 레이어를 넣지 않았습니다. 신뢰 가능한 네트워크에서만 열어 두는 것이 안전합니다.
- 기본 `codex` 실행 파일 경로는 `C:\nvm4w\nodejs\codex.cmd`입니다. 다르면 `web/.env.local`에서 `CODEX_BIN`을 바꾸면 됩니다.
- 볼트 루트는 코드에서 저장소 루트의 `obsidian/`로 고정되어 있습니다.

## 이 볼트에서 하지 않는 것

- 원본 자료를 임의로 수정하지 않습니다.
- 위키를 단순 채팅 기록 저장소로 쓰지 않습니다.
- 출처 없는 LLM 답변을 위키의 확정 지식으로 저장하지 않습니다.
